Aura Work supports 8 AI providers covering both cloud-hosted and local (Ollama, LM Studio) models. Each provider has its own adapter that handles authentication, model discovery, and chat completions. Providers marked as local never send data off your machine.

Architecture

How providers work

Each provider implements a ProviderAdapter interface with listModels(), validateCredentials(), and chat() methods. The system auto-discovers available models on connection and caches them. OpenAI-compatible providers share a single adapter implementation.

Credential security

API keys are encrypted using the device-bound vault before storage. The vault supports biometric unlock on supported platforms. Credentials are never logged or exposed to the agent.

Usage tracking

Every task records input/output token counts and estimated cost per model. The audit log maintains a permanent record of all provider interactions for accountability.

Setup

Setting up providers

Each provider requires an API key or host configuration. Configure them via environment variables or through Settings → Providers in the desktop app.

OpenAI

Obtain an API key from platform.openai.com.

# Environment variable
export OPENAI_API_KEY="sk-proj-xxxxxxxxxxxx"

# Verify connectivity
curl -s https://api.openai.com/v1/models \
 -H "Authorization: Bearer $OPENAI_API_KEY" | head -5

Adapter API

Provider adapter interface

ProviderAdapter protocol

Each provider implements the following interface:

interface ProviderAdapter {
 listModels(): Promise<ModelDescriptor[]>
 validateCredentials(): Promise<boolean>
 chat(request: ChatRequest): AsyncIterable<ChatChunk>
}

listModels — Returns available models from the provider, cached for 5 minutes after first fetch. Used to validate model names in configuration.

validateCredentials — Tests the API key or connection by making a lightweight request. Returns false if credentials are invalid or expired.

chat — Sends a chat completion request and streams the response. Supports tool calls and structured output when the provider supports them.

OpenAI-compatible adapter

Providers that use the OpenAI chat completions API format share a single adapter implementation. Configure the base URL and API key:

{
 "type": "openai-compatible",
 "base_url": "https://api.custom-provider.com/v1",
 "api_key_env": "CUSTOM_PROVIDER_KEY",
 "models": ["custom-model-1", "custom-model-2"]
}

Custom adapter example

To add a new provider, implement the adapter and register it in the provider registry:

import { ProviderAdapter, ChatRequest, ChatChunk } from '@aura/providers';

class MyCustomProvider implements ProviderAdapter {
 async listModels() { return [{ id: 'my-model', name: 'My Model' }]; }
 async validateCredentials() { return this.apiKey.startsWith('sk-'); }
 async *chat(request: ChatRequest) {
   const response = await fetch(this.baseUrl, { /* ... */ });
   for await (const chunk of response.body) yield chunk;
 }
}

registerProvider('my-provider', MyCustomProvider);

Local providers

Ollama: Install from ollama.ai, run ollama serve, then ollama pull llama3.2. Aura Work auto-discovers Ollama when the service is running on the default port.

LM Studio: Download from lmstudio.ai, start the local inference server (defaults to http://127.0.0.1:1234). Aura Work detects it automatically.

Custom Endpoint: Point to any OpenAI-compatible API endpoint. Set the URL in Settings → Providers → Custom Endpoint, or via AURA_CUSTOM_ENDPOINT environment variable.
